# 策略系统报告蓝图 V2

## 一、产品边界

系统面向用户的正式产出只保留五类：

1. 交易日盘前报告
2. 交易日 14:45 尾盘操作建议
3. 交易日最终盘后复盘
4. 每周日周度复盘
5. 每月最后一个自然日后的月度复盘

market_timing、theme_tracker、portfolio_review、stock_pool、buy_strategy、risk_control、chief_decision、各类 TDX 技能均为内部分析与证据层，不再默认作为用户必须阅读的独立报告。

核心原则：

> 数据和技能形成证据；RiskDecision 控制边界；ChiefDecision 统一动作；五类报告负责按时点表达。

---

## 二、统一数据层

### 1. 交易事实

- `master_trade_ledger.csv`：只增不改的全量交易主台账
- `daily_new_trades_template.csv`：每日新增成交输入
- `current_positions.json`：根据增量成交维护的持仓数量与成本
- 收盘市值、盈亏、仓位：必须以当日收盘价重估

### 2. 市场事实

- 大盘指数：上证、深证、创业板、科创、北证等
- 市场宽度、成交额、涨跌停、炸板率、连板高度
- 0AMV：使用跨日状态机；进入空头后，仅当后续单日涨幅严格大于 4% 才切换为做多
- 板块与持仓所属板块的实时/收盘走势
- 公告、政策、产业、海外市场和突发事件

### 3. 质量与时效

每个关键字段必须标记：`source`、`as_of`、`quality`、`confirmed`。

- 缺失或过期数据不得提高交易权限
- 盘前不能使用当日收盘数据
- 14:45 必须使用盘中实时数据
- 盘后必须使用当日收盘数据
- 用户未确认交易和 0AMV 时，不得生成“最终盘后复盘”

## 三、正式报告一：盘前报告

### 时间

交易日 08:30；确认交易日后运行。

### 信息窗口

上一个交易日收盘后至本交易日盘前。所有消息必须带发布时间和来源，旧闻不得冒充新催化。

### 必需输入

- 隔夜海外指数、商品、汇率和科技链表现
- 政策、监管和宏观消息
- A股公司公告、业绩预告、回购、减持、停复牌等
- 持仓和候选股相关公告
- 产业新闻、事件催化和风险事件
- 上一交易日最终市场、板块、持仓和 0AMV 有效状态
- 当前确认持仓和上一交易日生成的观察池/条件计划

### 固定结构

1. 隔夜与盘前核心结论
2. 重要消息、公告与可信来源
3. 对大盘及主要风格的影响路径
4. 对主线和重点板块的影响
5. 对每只持仓的影响
6. 今日风险清单与禁止动作
7. 开盘后 30 分钟验证点
8. 今日条件计划摘要（只列已通过风控的计划）

### 边界

盘前报告主要回答“今天可能受什么影响、开盘后验证什么”，不根据盘前消息直接下无条件买卖指令。

### 正式产物

`03_daily_plans/YYYY-MM-DD_premarket_report.md`

---

## 四、正式报告二：14:45 尾盘操作建议

### 时间

交易日 14:45。

### 必需输入

- 经确认或标记时效的当前持仓数量与成本
- 14:45 实时个股价格、盈亏和成交状态
- 当日主要指数、市场宽度、成交额、情绪和 0AMV 盘中状态
- 全市场重点板块走势
- 每只持仓所属板块、所属市场/风格指数走势
- 个股相对板块、相对指数强弱
- 盘前计划、当日实际交易和最新风险证据

### 固定结构

1. 14:45 市场许可与隔夜风险
2. 大盘、主要风格和市场情绪
3. 全市场重点板块及主线变化
4. 持仓逐只实时复核
5. 卖出优先级：清仓/止损/减仓/保护利润/持有
6. 补仓审核：仅限原计划、市场允许、板块支持、个股触发、风险未否决
7. 收盘前执行顺序与滑点/流动性约束
8. 数据时效与未确认项

### 动作边界

- 卖出和风险处理优先于补仓
- 波段仓的计划性卖出原则上在 14:45 后确认，避免盘中洗盘导致恐慌卖出；硬止损、重大风险和流动性恶化不等待尾盘
- 持仓曾形成有效浮盈后，必须检查最大浮盈、含费保本价和保本保护带；接近保本且尾盘未修复时优先退出，防止赢转亏
- 禁止因浮亏或 J 值低直接摊低成本
- 补仓必须有预先定义的 EntryRule、止损位和风险预算
- 新开仓和补仓必须严格等待已定义信号确认；信号未出现时只允许继续观察
- B池最多仅观察，C/D池不得形成可执行补仓/开仓动作
- 14:45 按 B1 默认盘中不交易，上一确认收盘持仓可作为盘中持仓基线；若用户告知或台账检测到目标日成交，则必须先更新持仓
- 持仓基线确认且当日全持仓行情完整时，可以输出精确减仓数量；不得据此放宽加仓或新开仓权限

### 正式产物

`03_daily_plans/YYYY-MM-DD_1445_action_report.md`

## 五、正式报告三：最终盘后复盘

### 触发

收盘后，且用户已提供：

- 当日新增交易记录，或明确“今日无交易”
- 当日收盘 0AMV
- 必要时提供计划外交易原因

### 执行顺序

1. 将新增成交追加至 `master_trade_ledger.csv`
2. 更新并确认持仓数量与成本
3. 写入 0AMV，并更新跨日有效状态
4. 获取当日收盘大盘、板块、持仓行情
5. 按收盘价重估持仓盈亏、市值和组合仓位
6. 对照盘前报告、14:45建议与实际成交
7. 生成最终复盘，并派生下一交易日观察池和条件计划

分析顺序必须遵循：

```text
新闻（信息/政策/风向/舆情）
-> 大盘与指数
-> 主线/主题/新旧热点
-> 持仓逻辑与买卖点
-> 自动/手动选股
-> 次日仓位和交易计划
```

### 固定结构

1. 今日计划、实际执行和纪律偏差
2. 新闻、政策、风向与舆情
3. 大盘、市场宽度、情绪、成交额、指数和0AMV状态
4. 主线、主题、新热点、持续性和过时热点
5. 持仓逐只逻辑、盘面关键点、买卖点及加减仓机会
6. 自动/手动选股结果及A/B/C/D分层
7. 次日总仓位、逐股买卖方向、比例/数量、触发条件和无效条件
8. 有效规则、失效规则、数据问题和风险提示

单票交易后仓位不得超过账户总权益的20%；若其他规则上限更低，执行更低值。精确买卖数量仍受数据时效门约束。详细流程见 `AFTER_CLOSE_REVIEW_WORKFLOW.md`。

### 正式产物

- `04_reviews/daily/YYYY-MM-DD_final_review.md`
- `04_reviews/daily/YYYY-MM-DD_final_review.json`

下一交易日候选池和条件计划属于该复盘的派生结构化产物，不单独作为第六份核心报告。

## 六、正式报告四：每周日复盘

### 时间范围

最近一个完整交易周。周日运行；如周日数据或交易记录未补齐，则标记阻塞，不生成伪完整报告。

### 核心目标

周复盘重点评估“执行纪律和短周期规则质量”，而不是简单拼接五份日报。

### 固定结构

1. 本周市场状态与主线演化
2. 周初计划与实际市场路径的偏差
3. 本周新增交易和持仓变化
4. 每笔交易的计划关联、执行价格和计划外原因
5. 持仓收益、已实现收益、浮动收益和组合回撤
6. 卖出、止损、补仓及仓位纪律
7. A/B/C/D池迁移和计划转化率
8. 本周有效/失效规则
9. 下周市场情景、风险预算和重点观察

### 核心指标

- 计划内交易占比
- 计划外交易数
- 止损合规率
- 建议动作执行率
- 单笔胜率、盈亏比、MFE/MAE（数据具备后）
- 周度收益和最大回撤
- A/B池进入交易的转化率
- 数据完整率和报告准时率

### 正式产物

`04_reviews/weekly/YYYY-Www_weekly_review.md/json`

---

## 七、正式报告五：月度复盘

### 时间范围

完整自然月；应在该月最后一个交易日数据和交易记录确认后生成。

### 核心目标

月复盘重点评估“策略是否有效、风险是否可控、规则是否需要升级”，不能只评价单笔交易。

### 固定结构

1. 月度市场环境和主线阶段
2. 月度收益、回撤、波动和资金使用效率
3. 已实现/未实现盈亏及个股、板块归因
4. 买入、补仓、卖出、止损和仓位规则表现
5. 计划内外交易与行为偏差
6. 选股池、主线识别和风险否决效果
7. 规则版本表现及样本量
8. 重复错误和重大风险事件
9. 下月策略参数、风险预算和观察方向

### 核心指标

- 月度收益率、最大回撤、收益回撤比
- 胜率、盈亏比、期望值
- 组合集中度、行业集中度和相关性暴露
- 平均持有期、换手率和费用
- 止损纪律、补仓纪律和仓位纪律
- 规则/策略版本贡献
- 与市场基准对比

### 正式产物

`04_reviews/monthly/YYYY-MM_monthly_review.md/json`

## 八、内部产物与正式报告的关系

内部结构化对象：

- MarketState
- NewsEvidence / NoticeEvidence
- SectorState
- HoldingReview
- StockCandidate
- EntryRule / ExitPlan
- PositionAdvice
- RiskDecision
- ChiefDecision

这些对象可以保留 JSON 和调试 Markdown，但默认不作为面向用户的独立日报推送。

报告渲染顺序：

```text
原始数据与消息
  → 标准化证据
  → 市场/板块/持仓/候选分析
  → RiskDecision
  → ChiefDecision
  → 对应时点正式报告
```

## 九、调度边界

- 08:50：预采集公告、海外行情和RSS候选，不生成正式报告
- 09:05：复用08:50采集结果，执行质量门、决策、渲染、校验和投递
- 14:45：采集尾盘行情，运行严格持仓检查并投递尾盘报告
- 15:15：只提醒补充新增交易和0AMV
- 20:30：用户补数后运行确定性盘后流水线和最终复盘
- 周日：周度复盘
- 月末最后一个交易日数据确认后：月度复盘

交易日历优先级：人工`overrides` > 交易所官方休市安排 > 本地通达信缓存 > 周末规则 > `unknown`。非交易日跳过并保持安静。

所有任务必须先检查交易日或统计周期完整性。任务失败应记录原因并告警，不得以空报告冒充成功。

## 十、持仓数据维护

### 唯一交易事实源

- `master_trade_ledger.csv`：只增不改的全量交易主台账
- `daily_new_trades_template.csv`：每日只补新增成交
- `current_positions.json`：由增量成交维护数量与成本

Excel 全量文件只用于历史迁移、对账和灾备重建，不再作为每日必需输入。详细规则见 `INCREMENTAL_TRADE_LEDGER.md`。

### 关键限制

1. 所有报告必须展示持仓导入时间
2. 14:45 默认沿用上一确认收盘持仓作为盘中基线
3. 14:45 的默认无交易基线不得放宽加仓或新开仓权限
4. 不允许用行情涨跌自行推测买卖数量或持仓变化
5. 用户盘后补充交易记录后，必须先重新导入，再做复盘

## 十一、盘后复盘六步流程

```text
新闻 -> 大盘 -> 热点 -> 持仓 -> 选股 -> 次日交易计划
```

前一层是后一层的过滤条件。不得先决定买卖，再倒推新闻、热点或指标作为理由。

### 报告质量三级门

- `blocked`：交易记录/无交易确认或当日0AMV任一核心事实缺失，不生成正式复盘
- `degraded`：核心事实齐全但辅助数据部分缺失，仍须生成完整六步复盘，逐项标注缺失
- `complete`：核心事实与所需行情、账户和分析数据均为当日已确认口径

### 单票20%硬上限

任何一只股票的交易后仓位不得超过账户总权益的20%。新买和加仓前必须计算交易后预计仓位；超过20%的计划自动否决或缩减数量。

## 十二、当前需要调整的旧设计

1. 将 Excel 全量导入模式降级为历史迁移/灾备，日常改用增量主台账。
2. 取消“统一日报”作为额外正式产出；其内容分别归入盘前、14:45和盘后报告。
3. `daily_plan` 不再作为独立第4份日内报告；ChiefDecision作为三份报告的内部决策对象。
4. 盘前增加消息时间窗、来源、可信度和持仓影响映射。
5. 14:45 从“报告骨架”升级为实时执行报告，明确卖出优先和补仓硬门槛。
6. 盘后复盘必须包含收盘大盘、板块、持仓板块和同日收盘重估。
7. 正式建立周/月复盘数据契约和指标，不再只保留空目录。
8. 所有报告增加唯一 `report_id`、数据截止时间、规则版本和输入清单，支持审计与重跑。
